# План тестирования проекта Student Progress Tracker

## 1. Введение

Данный документ описывает план тестирования учебного консольного Java-приложения **Student Progress Tracker**.  
Цель тестирования — проверить корректность работы основных функций приложения и выявить ошибки, а также неудобства при использовании.

Приложение позволяет:
- добавлять студентов;
- добавлять оценки студентам;
- просматривать список студентов и их оценок;
- рассчитывать средний балл;
- экспортировать данные в файл.

---

## 2. Цели тестирования

Основные цели тестирования:

1. Проверить, что все основные функции приложения работают в соответствии с требованиями:
   - добавление, просмотр и удаление студентов;
   - добавление и просмотр оценок;
   - расчёт среднего балла;
   - экспорт данных.

2. Выявить ошибки обработки некорректных и пустых данных (валидация ввода).

3. Оценить удобство взаимодействия пользователя с консольным интерфейсом:
   - понятность меню;
   - информативность сообщений;
   - предсказуемость поведения программы.

4. Зафиксировать найденные дефекты и предложения по улучшению в отдельном документе (`issues.md`).

---

## 3. Объект тестирования

**Объект тестирования** — консольное Java-приложение **Student Progress Tracker**, представляющее собой программу для учёта успеваемости студентов.

Основные тестируемые модули и функции:

- Модуль управления студентами:
  - добавление нового студента;
  - вывод списка студентов;
  - (при наличии) удаление студента.

- Модуль управления оценками:
  - добавление оценки студенту;
  - просмотр списка оценок студента.

- Модуль расчёта статистики:
  - расчёт среднего балла по студенту.

- Модуль работы с файлами:
  - экспорт данных в файл;
  - корректность формата экспортируемых данных.

- Консольный интерфейс:
  - структура меню;
  - сообщения об ошибках и подсказки.

---

## 4. Область и объём тестирования

### 4.1. Включено в тестирование

В рамках данного плана будут проверены:

- Основные пользовательские сценарии:
  - запуск приложения;
  - добавление студента;
  - добавление оценки;
  - просмотр списка студентов;
  - просмотр успеваемости по студенту;
  - расчёт среднего балла;
  - экспорт данных.

- Обработка некорректного ввода:
  - пустые значения;
  - неверные типы данных (например, буквы вместо чисел);
  - выход за допустимый диапазон оценок (например, 0 или 200, если ожидается 1–5).

- Базовая юзабилити:
  - понятность структуры меню;
  - логичность названий пунктов;
  - наличие/отсутствие подтверждений при потенциально опасных действиях (например, удаление).

### 4.2. Не включено в тестирование

В рамках данного учебного проекта **не тестируются**:

- Производительность при больших объёмах данных.
- Кроссплатформенные особенности (подробно).
- Безопасность (шифрование, защита от атак и т.п.).
- Интеграция с внешними системами (если они отсутствуют).

---

## 5. Типы тестирования

В рамках данного проекта будут использоваться следующие типы тестирования:

1. **Функциональное тестирование**
   - Проверка, что функции приложения работают согласно ожидаемому поведению и требованиям.
   - Фокус на корректности бизнес-логики: добавление, обработка и вывод данных.

2. **Тестирование удобства использования (юзабилити)**
   - Оценка простоты и понятности работы с приложением для новых пользователей.
   - Проверка интуитивности меню и сообщений.

3. **Smoke-тестирование (краткое)**
   - Проверка того, что приложение в принципе запускается, основное меню работает, и базовые сценарии (добавление студента, добавление оценки, просмотр) не приводят к критическим ошибкам.

---

## 6. Среда тестирования и инструменты

Тестирование проводится вручную (manual testing) в следующей среде:

- **Операционная система**: Windows 11  
- **JDK**: OpenJDK 17  
- **Среда разработки (IDE)**: IntelliJ IDEA Community (версия может быть указана, например: 2023.1)  
- **Способ запуска**:
  - через IDE (запуск `App.main()`),
  - либо через терминал с использованием команд:
    - `mvn clean install`
    - `java -jar target/student-progress-tracker-1.0.jar`

Дополнительные инструменты:

- Текстовый редактор для проверки содержимого экспортируемых файлов (`students-data.txt` или аналогичный).
- Таблица (Google Sheets / Excel) или markdown-файл `test_report.md` для фиксации результатов тест-кейсов.

---

## 7. Подход и стратегия тестирования

1. Выполнить **smoke-тестирование**:
   - запустить приложение;
   - убедиться, что главное меню отображается и не происходит аварийного завершения.

2. Провести **функциональное тестирование** по заранее составленным тест-кейсам:
   - позитивные сценарии (корректные данные);
   - негативные сценарии (пустой ввод, некорректные значения).

3. Провести **юзабилити-тестирование**:
   - дать приложение нескольким пользователям (одногруппникам);
   - наблюдать, насколько легко они находят нужные функции;
   - собрать их субъективную оценку удобства (дополнительно фиксируется через Google Forms, результаты — в `feedback_summary.md`).

4. Все найденные несоответствия и дефекты заносятся в:
   - `testing/test_report.md` — фактические результаты тест-кейсов;
   - `testing/issues.md` — список багов и предложений по улучшению.

---

## 8. Критерии начала и завершения тестирования

### 8.1. Критерии начала

Тестирование начинается при выполнении следующих условий:

- Приложение успешно компилируется без ошибок.
- Основной функционал реализован (минимальный набор функций доступен: добавление студентов, добавление оценок, просмотр данных).
- Есть возможность запуска приложения в тестовой среде.

### 8.2. Критерии завершения

Тестирование может быть завершено, когда:

- Все запланированные тест-кейсы выполнены и зафиксированы в `test_report.md`.
- Критические и высокоприоритетные дефекты либо устранены, либо задокументированы (`issues.md`) и согласованы как допустимые для учебной версии.
- Собрана и проанализирована пользовательская обратная связь, подготовлен файл `feedback_summary.md`.
- В README добавлен раздел «Тестирование и обратная связь» с ссылками на отчёты.

---

## 9. Риски и ограничения

Основные риски и ограничения:

- Ограниченное время на разработку и тестирование (учебный проект).
- Небольшое количество тестировщиков (ограниченный набор устройств и сценариев).
- Возможные неточности в требованиях, так как проект учебный и может изменяться в процессе разработки.

---

## 10. Артефакты тестирования

В результате выполнения плана тестирования будут подготовлены следующие артефакты:

- `testing/test_plan.md` — данный документ (план тестирования).
- `testing/test_report.md` — таблица с тест-кейсами и фактическими результатами.
- `testing/feedback_summary.md` — анализ отзывов пользователей по результатам опроса.
- `testing/issues.md` — список найденных ошибок и предложений по улучшению.
